## 2024-07-15 - [Service Worker State Loss in Chrome vs Firefox]
**Learning:** Chrome Manifest V3 uses Service Workers for backgrounds, which can suspend and lose in-memory variable state (like `Set` or `Map`) mid-execution or between events. Firefox Manifest V3 uses Event Pages (via `scripts` array), which also can suspend, but their lifecycles may differ or users might notice state loss differently. In this extension, storing `activeDownloadIds` in a memory `Set` meant cancelations would fail if the Service Worker slept. Also, unreturned async operations in `onMessage` can cause the worker to suspend before completing.
**Action:** Always store extension state that needs to persist across events in `browser.storage.local` or `browser.storage.session`, never in memory variables. Always return the Promises for long-running async tasks inside `onMessage` listeners so the browser knows to keep the background context alive until they resolve.
## 2024-07-15 - [Concurrency and Arrays in browser.storage.local]
**Learning:** `browser.storage.local.get` and `set` are asynchronous. Reading an array, appending an item, and writing it back creates a race condition when multiple asynchronous tasks (like overlapping downloads finishing in `browser.downloads.onChanged`) do this simultaneously. They will overwrite each other's changes.
**Action:** Do not use arrays in `browser.storage.local` to track lists of independent items being concurrently updated. Instead, store each item under its own independent storage key (e.g., `set({ [\`dl_${id}\`]: true })`).
